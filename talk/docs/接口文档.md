# 项目结构与接口文档

## 1. 项目结构检查结果

### 1.1 技术栈

- 后端：Spring Boot 3.4.3
- 语言：Java 17
- ORM：MyBatis-Plus
- 关系型存储：SQLite
- AI 能力：Spring AI
- 向量存储：Qdrant / Spring AI `VectorStore`
- 前端：`qianduan/` 下的 Vite + React + TypeScript

### 1.2 目录结构

```text
talk/
├─ src/main/java/com/example/memory
│  ├─ ai                  # AI 访问门面
│  ├─ common              # 统一返回体与返回码
│  ├─ config              # AI、聊天、RAG、向量配置
│  ├─ controller          # REST 接口入口
│  ├─ dto                 # 请求对象与校验规则
│  ├─ entity              # SQLite 实体
│  ├─ exception           # 业务异常与全局异常处理
│  ├─ mapper              # MyBatis-Plus Mapper
│  ├─ service             # 聊天编排、记忆提取、画像、时间线、向量检索
│  └─ vo                  # 响应对象
├─ src/main/resources
│  ├─ application.yml     # 服务与 AI 配置
│  └─ schema.sql          # SQLite 建表脚本
├─ src/test/java          # 控制器、服务、配置集成测试
├─ qianduan/              # 前端页面与组件
├─ memory-backend.db      # 本地 SQLite 数据库
└─ docs/接口文档.md       # 当前文档
```

### 1.3 后端模块职责

- `controller`
  - 暴露健康检查、AI 测试、普通聊天、RAG 聊天、记忆管理、向量检索、用户画像、时间线接口。
- `service`
  - `ChatOrchestrationService`：普通对话，写入 `message_event`。
  - `RagChatService`：先检索向量记忆，再拼装提示词进行回复。
  - `MemoryExtractionService`：从最近用户消息中规则提取长期记忆。
  - `MemoryRecordService`：记忆创建、查询、去重更新，并自动同步到向量库。
  - `VectorMemoryService`：记忆向量写入、相似度检索、RAG 检索。
  - `UserProfileService` / `UserTimelineService`：基于历史记忆与事件聚合用户视图。
- `entity + schema.sql`
  - 核心表为 `user_account`、`project_space`、`chat_session`、`message_event`、`memory_record`、`conversation_summary`。

### 1.4 当前项目特点

- 使用统一响应格式 `Result<T>`，默认结构为 `code`、`message`、`data`。
- 聊天历史真实来源是 SQLite 的 `message_event`，不是 Spring AI 内存。
- 记忆有三层作用域：`global`、`project`、`chat`。
- 创建或更新 `memory_record` 后会自动执行向量写入。
- RAG 检索默认优先当前聊天、当前项目，再补充全局记忆。
- 当前没有 OpenAPI/Swagger 自动文档，接口说明需要以代码为准维护。

## 2. 统一约定

### 2.1 基础地址

- 默认端口：`8080`
- 建议本地基础地址：`http://localhost:8080`

### 2.2 统一返回格式

成功：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

校验失败：

```json
{
  "code": 422,
  "message": "fieldName: error message",
  "data": null
}
```

业务错误：

```json
{
  "code": 400,
  "message": "business error message",
  "data": null
}
```

系统异常：

```json
{
  "code": 500,
  "message": "internal server error",
  "data": null
}
```

### 2.3 通用错误码

| code | message | 说明 |
|---|---|---|
| 200 | success | 请求成功 |
| 400 | bad request | 业务校验失败，如项目或会话不存在 |
| 422 | validation failed | 参数校验失败 |
| 500 | internal server error | 未处理异常 |

## 3.3 普通聊天

### POST `/api/chat/send`

说明：
- 普通对话接口。
- 会先校验 `projectId` 与 `chatId` 的归属关系。
- 用户消息与模型回复都会写入 `message_event`。
- 会从当前会话最近若干条消息中构造上下文，默认上限见 `app.chat.context-message-limit`。

请求体：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| projectId | long | 是 | 项目 ID |
| chatId | long | 是 | 会话 ID |
| message | string | 是 | 用户消息 |

请求示例：

```json
{
  "projectId": 1,
  "chatId": 1,
  "message": "帮我总结一下当前项目的目标"
}
```

响应 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| projectId | long | 项目 ID |
| chatId | long | 会话 ID |
| userMessageId | long | 新写入的用户消息事件 ID |
| assistantMessageId | long | 新写入的助手消息事件 ID |
| assistantReply | string | 模型回复 |
| usedContextMessageCount | integer | 本次参与上下文的历史消息数 |
| userSequenceNo | integer | 用户消息序号 |
| assistantSequenceNo | integer | 助手消息序号 |
| provider | string | 模型提供方 |
| model | string | 模型名 |

常见业务错误：
- `project space not found`
- `chat session not found`
- `chat session does not belong to project space`

## 3.5 记忆管理

- 手动创建一条记忆。
- 创建成功后会自动同步到向量库。

请求体：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| userId | long | 是 | 用户 ID |
| projectId | long | 是 | 项目 ID |
| chatId | long | 否 | 会话 ID |
| sourceMessageEventId | long | 否 | 来源消息事件 ID |
| scopeType | string | 是 | `global/project/chat` |
| memoryType | string | 是 | 记忆类型枚举 |
| title | string | 否 | 标题 |
| summary | string | 是 | 摘要 |
| detailText | string | 否 | 详情文本 |

| metadataJson | string | 否 | 元数据 JSON 字符串 |

| chatId | long | 会话 ID |

| scopeType | string | 作用域 |

| title | string | 标题 |
| summary | string | 摘要 |
| detailText | string | 详情 |

### POST `/api/chat/{chatId}/extract-memory`

说明：
- 从当前聊天最近若干条用户消息中抽取记忆。
- 仅扫描 `USER` 角色消息。
- 当前实现为规则抽取，不是大模型抽取。
- 抽取出的记忆会执行去重更新，并同步到向量库。

路径参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| chatId | long | 是 | 会话 ID，必须大于 0 |

请求体：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| projectId | long | 是 | 所属项目 ID |

响应 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| projectId | long | 项目 ID |
| chatId | long | 会话 ID |
| scannedMessageCount | integer | 本次扫描消息数 |
| extractedCount | integer | 成功提取记忆数 |
| memories | array | 提取出的记忆列表 |

规则说明：
- 决策类关键词优先级最高，如“决定”“最终”“确认”。
- 约束类关键词如“必须”“不允许”“只能”“禁止”。
- 需求类关键词如“需要”“要求”。
- 偏好、习惯、画像也会根据关键词推断。
- 包含“以后”“长期”“通常”等内容时更可能被判为 `global` 记忆。
- 未命中的文本不会生成记忆。

## 3.6 向量记忆

### POST `/api/vector/memory/upsert/{memoryId}`

说明：
- 将指定记忆写入向量库。
- 如果该记忆已存在，行为由底层向量库 upsert 语义决定。

路径参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| memoryId | long | 是 | 记忆 ID，必须大于 0 |

响应 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| memoryId | long | 记忆 ID |
| documentId | string | 向量文档 ID，当前等于记忆 ID |
| scopeType | string | 作用域 |
| memoryType | string | 记忆类型 |
| status | string | 当前固定为 `UPSERTED` |

常见业务错误：
- `memory record not found`

### POST `/api/vector/memory/search`

说明：
- 对已写入向量库的记忆做相似度检索。
- 先查当前聊天，再查当前项目，结果不足时再补充其他项目的全局类记忆。

请求体：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| userId | long | 是 | 用户 ID |
| projectId | long | 是 | 当前项目 ID |
| chatId | long | 否 | 当前会话 ID |
| scopeTypes | string[] | 否 | 过滤的作用域列表，仅允许 `global/project/chat` |
| queryText | string | 是 | 检索文本 |
| topK | integer | 否 | 返回数量，最小值 1；为空时取配置默认值 |

请求示例：

```json
{
  "userId": 1,
  "projectId": 1,
  "chatId": 1,
  "scopeTypes": ["project", "global"],
  "queryText": "用户对技术栈有什么要求",
  "topK": 5
}
```

响应 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| queryText | string | 原始查询文本 |
| topK | integer | 实际返回上限 |
| totalHits | integer | 总命中数 |
| projectHits | integer | 当前项目范围命中数 |
| supplementalHits | integer | 补充范围命中数 |
| hits | array | 命中列表，元素为 `VectorMemoryHitVO` |

注意：
- 当前补充范围判断逻辑是“全局类 scope”，代码中还兼容了 `user` 和 `global_*` 前缀，但当前 DTO 校验仅允许 `global/project/chat`。
- 若 Qdrant 未启用或未正确连接，实际运行时可能由底层向量实现抛出异常。

## 4. 调用顺序建议

如果要跑通一个完整业务流程，建议顺序如下：

1. 先准备基础数据：`user_account`、`project_space`、`chat_session`
2. 调用 `POST /api/chat/send` 写入聊天消息
3. 调用 `POST /api/chat/{chatId}/extract-memory` 抽取长期记忆
4. 调用 `POST /api/vector/memory/search` 验证向量检索效果
5. 调用 `POST /api/chat/rag-send` 验证带记忆增强的对话
6. 调用 `GET /api/profile/{userId}` 和 `GET /api/timeline/{userId}` 查看聚合结果

## 5. 结构检查结论

- 后端分层清晰，`controller -> service -> mapper -> SQLite/Qdrant` 路径明确。
- 接口定义集中，当前总计 13 个对外 REST 接口。
- 请求参数基本都有 Bean Validation，统一异常处理已接入。
- 接口文档目前未自动生成，建议后续补充 Swagger/OpenAPI 或在 CI 中校验文档与 DTO 一致性。
- 当前前端目录已存在，但后端没有看到显式的跨域配置，前后端联调时需要确认代理或 CORS 方案。
